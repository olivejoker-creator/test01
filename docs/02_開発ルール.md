# 02_開発ルール

> このファイルの役割：
> Minecraft MOD制作を毎回同じ流れで進行するための共通ルールを定義する。
> 本プロジェクトでは、ChatGPT・Claude・Claude Code・Codexを役割分担して使用する。
> 各担当は定められた範囲のみを担当し、勝手に他工程を変更してはならない。

---

# 1. 基本方針

本プロジェクトは以下の順序で進行する。

```text
企画
↓
仕様書作成
↓
実装計画作成
↓
プログラム実装
↓
コンセプトアート作成
↓
アセット制作
↓
統合作業
↓
デバッグ
↓
最終確認
```

工程を飛ばしてはならない。

# 4. リポジトリ構成

このリポジトリ直下は、どのmodでも同じ構成にする。
modごとに変わるのは `<modid>` の中身だけで、入れ物（フォルダの形）は固定する。
Claude Codeは、ファイルの作成・配置をこの構成に従って行うこと。

## 標準構成

```text
<リポジトリ直下>/
├── docs/                         管理ドキュメント（00〜07）をここに集約
│   ├── 00_プロジェクト基盤定義.md
│   ├── 01_企画書.md
│   ├── 02_開発ルール.md
│   ├── 03_仕様書.md
│   ├── 04_実装計画.md
│   ├── 05_実装報告.md
│   ├── 06_デバッグ報告.md
│   └── 07_完成レビュー.md
├── src/                          MOD本体（Claude Code が Task001 で生成）
│   └── main/
│       ├── java/                 Javaコード
│       └── resources/
│           ├── META-INF/         mods.toml 等のメタデータ
│           ├── assets/<modid>/   ゲームに読み込まれる完成テクスチャ・モデル・lang・音
│           └── data/<modid>/     レシピ・ドロップ・タグ等（DataPack）
├── blockbench/                   モデルの元ファイル（.bbmodel 編集用オリジナル）
├── textures/                     テクスチャの元ファイル（編集用オリジナル）
├── animations/                   アニメーションの元データ
├── .github/                      Issue / PR テンプレート（必ずリポジトリ直下）
├── README.md
├── LICENSE
└── .gitignore
```

## 配置ルール

* 管理ドキュメント（00〜07）は必ず `docs/` 直下に置く。リポジトリ直下にバラ置きしない
* `<modid>` は `00_プロジェクト基盤定義` で決めた modID に置換する（modごとに変わるのはここだけ）
* 「編集用の原本」と「ゲームに読み込まれる完成品」を分ける

```text
原本（編集用）   … blockbench/ ・ textures/ ・ animations/
完成品（反映用） … src/main/resources/assets/<modid>/
```

* テクスチャ・モデルは `src/main/resources/assets/<modid>/` に入れて初めてゲームに反映される
* `.github/` は必ずリポジトリ直下に置く（1階層深いとGitHubが認識しない）
* `build/` ・ `run/` ・ `.gradle/` ・ `node_modules/` などの生成物は `.gitignore` で除外し、コミットしない
* この構成から外れる配置が必要になった場合は、勝手に変えず質問事項として記録する

# 5. GitHub運用ルール

使用ブランチ

```text
main
リリース専用

dev
日常開発用
```

ルール

* 開発は必ずdevで行う
* mainへ直接作業しない
* 仕様変更前にコミットする
* 作業完了後にコミットする
* コミットメッセージを記載する

例

```text
feat: add ruby sword

fix: recipe bug

docs: update specification

refactor: optimize entity logic
```

# 10. 複数PCでの運用ルール（2台以上で同じmodを触る場合）

同じリポジトリを複数のPCから編集する場合、変更の衝突（コンフリクト）を防ぐため必ず守る。

## 黄金ルール
- 作業を始める前：`git pull`（相手PCの最新を取り込む）
- 作業を終えたら：`git push`（自分の変更を上げる）
- 2台で同時に触らない（必ず片方ずつ・交互に）

## 流れ
1. PCで作業を始める前に `git pull`
2. 作業する（Claude Code / 取り込み / 直接編集）
3. 席を立つ前に必ず `git push`
4. もう一方のPCでは、始める前に `git pull` してから作業

## 注意
- push を忘れて別PCに移らない（「座ったら pull、立つ前に push」を習慣に）
- どちらが最新か迷ったら GitHub（クラウド）が正。作業前に pull すれば揃う
- push が衝突（rejected / non-fast-forward）したら、Claude Code に
  「git pull してコンフリクトを解消し、変更を消さないよう安全にマージして push し直して」と頼む
- ファイル編集はできるだけ Claude Code に寄せる（pull→作業→push を自動で行うため事故が減る）
- 各PCに Git・Node.js・Claude Code を個別に導入し、初回に認証する（設定は引き継がれない）
